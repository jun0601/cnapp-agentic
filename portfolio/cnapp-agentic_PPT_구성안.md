# cnapp-agentic 포트폴리오 PPT 구성안 (v6 — 공동 작업용으로 역할 배지 균형화)

> v6에서 바뀐 것 (준형 반영): **이 PPT는 2인이 함께 쓰는 통일된 한 벌**이다 — 완성 후 **표지(1번)와 트러블슈팅(16번)만 각자 버전으로 갈라지고**, 나머지 전 슬라이드는 끝까지 공동 자료로 유지한다. v5가 진우 관점에서 짜이면서 4·7·10·15번에 "★진우" 배지만 붙고 준형 담당분(Evidence·Triage, CSPM(AWS)·IAM Access Analyzer, attack-path 데이터 모델, CI/CD·공유인프라 등)엔 배지가 없어 한쪽으로 쏠려 있었던 걸 양쪽 다 표시하도록 균형을 맞췄다. 추가로 **RAG가 역할분담표엔 있는데 정작 슬라이드가 하나도 없어서 신설**(신 9번) — 대신 페이지 수가 늘지 않게 **"계약 기반 설계"는 독립 슬라이드를 없애고 4번(역할 분담)에 압축해 접어 넣었다**(2인 병렬개발을 가능케 한 메커니즘이라 역할분담 바로 옆이 논리적으로도 더 맞음). 총 슬라이드 수는 19장 그대로. 나머지 구조·문구·디자인 원칙은 v5를 그대로 유지(잘 짜여 있음, 손댈 이유 없음).
>
> v4→v5 변경 이력(진우, 참고용 보존):
> ① 개요의 "핵심 임팩트 미리보기"를 삭제하고 결과 섹션의 "성과"로 이동했다 — 숫자를 발표 초반에 스포일러하지 않고, 트러블슈팅(문제 해결력)을 보여준 다음 마지막에 "그래서 이런 성과가 나왔다"로 마무리하는 게 더 설득력 있는 구조다.
> ② 개요의 "담당 영역 — 내가 만든 범위"를 "역할 분담"으로 이름을 바꾸고 위치를 2번으로 앞당겼다 — 2인 협업 프로젝트라는 걸 아키텍처를 보기 *전에* 먼저 밝힌다.
> ③ 구현의 "CI/CD & 운영 관측"을 "CI/CD"와 "운영 관측" 2개 슬라이드로 분리했다.

## 설계 원칙

1. **목차를 4개 섹션 + 하위 목록으로 구성한다.** 각 섹션 번호(01~04)를 모든 슬라이드 상단 kicker에도 동일하게 표시해서, 지금 몇 번 섹션을 보고 있는지 계속 알 수 있게 한다.
2. **"역할 분담" 슬라이드는 01 개요의 2번째 항목으로 앞당긴다.** 문제 정의 다음, 아키텍처를 보기 전에 "이건 2인 협업이고 이렇게 나눠서 했다"를 먼저 밝힌다 — 그래야 뒤이어 나오는 아키텍처·구현 슬라이드를 볼 때부터 채용담당자가 "이 부분이 이 사람 담당이구나"를 계속 의식하며 본다.
3. **"성과" 슬라이드(숫자 임팩트)는 04 결과의 첫 항목으로 배치한다.** 발표 초반에 숫자를 스포일러하지 않고, 트러블슈팅까지 다 보여준 다음 "그래서 결과가 이렇다"로 마무리하는 게 더 설득력 있다.
4. **트러블슈팅 3건(16번)과 표지(1번)만 완성 후 각자 버전으로 분리한다.** 지금 이 문서에 적힌 내용은 두 버전이 갈라지기 전 공용 뼈대이고, 이 두 슬라이드는 각자 `troubleshooting.md`·`CLAUDE.md` 변경 로그의 작성자 태그(`진우`/`준형`) 기준으로 본인 실작업만 골라 개별 작성한다.
5. **"우리가 만들었다" 톤과 "내가 했다" 톤을 구분한다.** 나머지 전 슬라이드(01~04, 표지·트러블슈팅 제외)는 프로젝트 전체(우리) 톤으로 통일한다 — 개인(나) 톤은 표지·트러블슈팅 두 곳에만 쓴다. 01의 역할 분담·02의 각 담당 배지(★준형/★진우)는 "누가 만들었는지" 사실을 표시하는 것이지 개인 홍보 톤은 아니다.
6. **구현의 "FinOps 비용 전략"과 결과의 "비용"은 역할이 다르다 — 겹치지 않게 쓴다.** 전자는 *어떻게* 설계했는지(전략·원칙), 후자는 *실제로 얼마가 나왔는지*(측정된 숫자)를 다룬다.

### 3. 문제 정의 + 우리의 답
상단에 **프로젝트 시작 계기 1줄**을 먼저 두고, 그 아래 좌(문제) → 우(해결 키워드) **1:1 매핑 구조**(4행)를 배치한다. 문제 4개는 "통합(CNAPP) → 상관(Attack Path) → 확장(Cross-Cloud) → 자동화(Agentic AI)" 순으로, 점점 넓어지는 스코프를 따라간다 — 뒤이어 "02 구현" 섹션이 이 4개 키워드를 같은 순서로 증명하는 구조이므로, 이 슬라이드가 곧 발표 전체의 예고편 역할을 한다.

**시작 계기(슬라이드 최상단, 작은 글씨 1줄):**
> "클라우드 보안 운영은 이미 개별 서비스 점검(CSPM)을 넘어 멀티클라우드 통합과 AI 에이전트 기반 자동 조사로 이동하고 있다 — 이 흐름을 설계부터 배포까지 직접 구현하며 검증해보고자 시작했다."

- 디자인 팁: 이 문장은 슬라이드 제목 바로 아래, 본문 표보다 한 단계 작은 글씨(캡션 톤)로 배치해서 "이건 배경 설명이고, 진짜 본문은 아래 표"라는 위계를 준다.

**문제 → 해결 키워드 매핑(4행):**

| # | 문제 | → | 해결 키워드 |
|---|---|---|---|
| 1 | **보안 신호 파편화** — Security Hub(ASFF)·Prowler(OCSF)·Trivy(trivy-json) 등 스캐너마다 출력 스키마와 심각도 기준이 전부 달라, 담당자가 여러 콘솔을 오가며 수동으로 취합·재판단해야 함 | → | **CNAPP** — CSPM·CIEM·취약점·KSPM·데이터·attack-path 6개 기둥을 OCSF-lite 공통 스키마(JSON Schema)로 정규화. 스캐너가 몇 개든 같은 `resource_id`·`control_id` 체계로 들어와 하나의 콘솔·하나의 우선순위 기준으로 통합 |
| 2 | **개별 finding의 함정** — Critical 취약점 1건과 Low 등급 IAM 과도권한 1건을 따로 보면 둘 다 급하지 않아 보이지만, 실제로는 전자로 침투해 후자로 측면이동하는 하나의 공격 체인일 수 있음. finding을 개별 심각도로만 나열하면 이 연결 관계 자체가 안 보임 | → | **Attack Path** — 침투 진입점→권한 확대→자격증명 탈취→데이터 유출로 이어지는 상관 규칙으로 finding들을 노드·엣지 그래프로 연결. 체인이 3단계 이상 이어지면 severity를 Critical로 자동 격상해, 개별로는 안 급했던 finding들이 실제로 이어지면 최우선순위로 끌어올림 |
| 3 | **클라우드 단일 도구의 사각지대** — AWS 전용 도구는 AWS만, Azure 전용 도구는 Azure만 볼 수 있어, AWS → Azure로 이어지는 경계 횡단 공격의 전체 경로를 파악 못함 | → | **Cross-Cloud** — AWS에 평문으로 남은 Azure 자격증명을 발견하면 그 경로를 "경계 횡단" 엣지로 별도 표시해 그래프 상에서 바로 구분. Azure 쪽은 Prowler로 실 Entra 테넌트를 스캔해 과도권한 앱·위험 consent 등 실 finding까지 확보 |
| 4 | **대화형 챗봇의 한계** — 기존 "AI 보안" 도구 대부분은 이미 수집된 데이터를 요약해 답하는 챗봇에 머문다. "이 버킷이 지금 실제로 공개돼 있는지"는 분석가가 직접 AWS 콘솔에 로그인해 read-only API를 하나씩 확인해야 함 | → | **Agentic AI** — Orchestrator→Triage→Hypothesis→Evidence→Reasoning 5단계 루프에서, Evidence 단계의 Bedrock이 계약으로 고정된 read-only API allowlist 안에서 스스로 도구를 골라 호출. 실제로 `GetBucketPolicy`·`GetPublicAccessBlock`을 호출해 실 S3 응답을 근거로 CONFIRMED 판정까지 실증 |

- 디자인 팁: 4행을 위에서 아래로 나열하고, 화살표(→) 부분에 시안(#06B6D4) 포인트 컬러를 줘서 시선이 자연스럽게 오른쪽으로 흐르게 한다.

### 4. 역할 분담
> 이 프로젝트는 **김진우·이준형 2인 협업**이다. 아래는 README §5 역할 분담표 원본 — 8개 영역을 반으로 나눠 서로의 영역까지 완전히 이해하는 방식으로 진행했다.

2열 표(준형 | 진우) — **양쪽을 대칭으로 표시**(어느 한쪽만 강조하지 않음, 8개 영역을 정확히 반씩 나눠 각자 상대 영역까지 이해하며 진행했다는 걸 보여주는 게 목적):

| 영역 | 준형 | 진우 |
|---|---|---|
| 앱 | 타깃 앱 + 관제 앱 | — |
| 토대 | CI/CD·Shift-Left·공유인프라 주도 | 모니터링·관제·추적(Grafana·CloudTrail·X-Ray) |
| 스캐너-CSPM | CSPM(Config·Prowler·Security Hub·Macie) | 워크로드(Trivy·kube-bench)+Azure(Entra CIEM) |
| 스캐너-CIEM | IAM Access Analyzer(AWS) | Entra ID(Azure) |
| 수집·정규화 | 수집부(EventBridge→SQS) | 정규화부(Lambda→OCSF-lite) |
| 엔진(Bedrock) | Evidence(tool use)·Triage | Hypothesis·Reasoning·Orchestrator |
| RAG | 코퍼스·임베딩·pgvector 적재 | 검색·LLM 답변 생성 |
| attack-path | 그래프 데이터 모델 | 상관 로직(5개 규칙 체인)·내러티브 |

- 하단 문구: "8개 영역을 정확히 반으로 나누되, 서로의 영역까지 완전히 이해하는 방식으로 진행했습니다 — 이어지는 02 구현에서 각 담당자를 ★준형/★진우 배지로 계속 표시합니다."
- 디자인 팁: 준형 열은 옅은 네이비, 진우 열은 옅은 시안처럼 **두 열을 대칭 색상으로 구분**해서 "정확히 반반 협업"이라는 인상을 준다(한쪽 열만 채우지 않기).

**이 표 바로 아래, 작은 보조 스트립으로 "어떻게 안 부딪히고 반씩 나눴나"를 압축해 붙인다** (원래 별도 슬라이드였던 "계약 기반 설계"를 페이지 수 유지를 위해 이 슬라이드에 접어 넣음 — 표가 "무엇을 나눴는지"라면 이 스트립은 "어떻게 안 부딪혔는지"라 바로 아래 붙는 게 논리적으로 자연스럽다):
- 가로 4칸 소형 배지 스트립: `JSON Schema 7종` · `INTERNAL Control 14종` · `CI Validate 게이트(4-assert)` · `mock=False 한 줄 스왑`
- 캡션 1줄: "컴포넌트 이음새를 JSON Schema로 먼저 고정 → 2인이 직렬 의존 없이 병렬 개발"
- 디자인 팁: 표보다 눈에 덜 띄게 — 아이콘+짧은 라벨만 있는 작은 칩 4개를 한 줄에, 표 하단에 옅은 구분선을 긋고 그 아래 배치.

### 5. 전체 아키텍처

### 6. 골든 시나리오 — 크로스클라우드 공격 경로
- 4단계 타임라인:
  1. **[AWS]** 워크로드 침투 — product 취약 이미지(KEV) → 컨테이너 장악, privileged로 측면이동
  2. **[AWS]** 자격증명 발견 — order 과도 IRSA + 평문 Azure SP 자격증명 획득
  3. **[AWS]** 데이터 탈취 — member 공개 S3에서 PII 탈취(Macie가 미리 탐지한 미끼)
  4. **[Azure]** 신원 장악 — 탈취 자격증명으로 Entra 과도권한 앱 장악 → 디렉터리 전체 통제
- 강조 문구: "③→④ 구간 = 크로스클라우드 경계(credential_theft) — 단일 클라우드 도구는 이 경계 횡단을 볼 수 없다"
- 디자인 팁: 3→4 화살표만 빨간색으로 강조

### 7. 에이전틱 엔진 — 5단계 능동 조사 루프
- Orchestrator(전체 지휘, ★진우) → Triage(비용통제 게이트, ★준형) → Hypothesis(공격 가설 생성, ★진우) → **Evidence(AI가 스스로 read-only API 호출, ★준형)** → Reasoning(판정+내러티브 생성, ★진우)
- 5단계 전체에 ★ 표시로 담당자를 표시 — Evidence·Triage=★준형, Orchestrator·Hypothesis·Reasoning=★진우 — 4번 슬라이드(역할 분담)의 "엔진(Bedrock)" 행과 그대로 연결
- 강조 배지: "✔ 챗봇 탈출의 단일 기준"

### 8. 실증: Bedrock 능동조사 증거
- 왼쪽 숫자 강조: 100%(판정신뢰도) · ~$0(검증비용) · E2E(배포→조사→판정→정리 전 구간 관통)
- 보조 설명: "Bedrock Haiku가 `s3:GetBucketPolicy`·`GetPublicAccessBlock`을 스스로 호출 → 실 S3 응답 근거로 판정"
- 시각자료: **스크린샷 필수** — Finding 상세의 능동조사(Evidence) 탭

### 9. RAG — 지식베이스 기반 설명·챗봇
> (신설) 역할분담표엔 RAG 행이 버젓이 있는데 정작 단독 슬라이드가 없었던 걸 채운다 — Evidence(8번)가 "능동 조사"라면 RAG는 "근거 기반 설명·질의응답"으로, 둘 다 Bedrock을 쓰지만 서로 다른 별개 파이프라인임을 명확히 한다.
- 흐름 다이어그램(왼→오): finding/질문 → Titan Embed v2(임베딩) → pgvector cosine 검색(코퍼스 14 control·24청크) → Bedrock이 검색된 근거로 답변 생성
- 2가지 실사용처: ① Finding 상세의 "설명" 탭(control_id 기반 자동 근거 설명) ② `/chat` AI 어시스턴트(자유 질의응답, 근거 chunk 인용)
- 강조 문구: "Evidence(조사 루프)와는 별개 파이프라인 — 같은 Bedrock을 쓰지만 판정이 아니라 설명·검색을 담당"
- 시각자료: **스크린샷 권장** — `/chat` 화면 또는 finding 설명 탭

### 10. 관제 콘솔 — 9개 화면 라이브 운영 중
- 2×2 스크린샷 그리드(전체 9화면 중 대표 4종 — 제목·캡션에 "9화면 중 대표 4종"으로 명시해 제목의 "9화면"과 실제 그리드 장수 불일치를 미리 해소):

  2. Attack-path 그래프 — AWS/Azure 레인 + 크로스클라우드 빨강 엣지
  3. Findings 목록 — 필터·AI 상태 컬럼
  4. AI 어시스턴트/시스템 — RAG 챗(/chat)·Bedrock 24h 사용량(/system)

### 16. 트러블슈팅 — 내가 직접 부딪힌 문제 3건 (★ 완성 후 각자 버전으로 분리 — 아래는 진우 초안, 준형은 `troubleshooting.md`·변경 로그의 "준형" 작성자 태그 기준 본인 사례 3건으로 별도 사본 작성)
> 아래 3건은 모두 `troubleshooting.md`·변경 로그에 **작성자 "진우"로 기록된** 본인 실작업이다.

1. **Prowler 실계정 연동 시 포맷 불일치 발견 (2026-07-03)** — CLI는 OCSF 형식으로 스캔을 실행하면서, 코드는 다른 포맷("prowler-json")으로 태깅해 정규화 파이프라인에 전달하는 불일치를 발견. mock 기반 회귀 테스트는 포맷을 직접 지정해 우회했던 탓에 안 드러났던 문제 — 실계정 연동 직전 전체 코드 재검증 과정에서 스캐너와 정규화 로직을 나란히 대조해 찾아내고 수정.
2. **Grafana 대시보드 "No data" — 3중 원인 규명 (2026-07-06)** — CloudWatch 패널이 계속 데이터를 못 불러오는 증상을, Grafana API로 쿼리를 직접 재현 실행해가며 3개 층위의 원인을 순서대로 특정: ① 쿼리 모드 판별 필드 누락으로 프론트가 자동실행을 안 함 ② 비용 계산(metric math) 쿼리는 반드시 Code 모드여야 함(Builder 모드는 400 에러) ③ Cognito 로그인 지표명이 문서상 이름과 실제 발행 이름이 달라 발생 — `list-metrics`로 실측 확인 후 3건 모두 수정.
3. **X-Ray 분산추적의 자기 회귀 발견·수정 (2026-07-07)** — 전날 직접 구현한 X-Ray 계측 코드가 Lambda의 읽기전용 세그먼트를 직접 변경하려다 예외(`FacadeSegmentMutationException`)를 던져, **정규화 이후 2-pass 파이프라인 트리거가 매번 조용히 실패**하고 있던 것을 실 CloudWatch 로그로 직접 찾아내 수정. 같은 검증 과정에서 "EventBridge는 트레이스가 자동 전파 안 된다"던 전날 자신의 설계 가정이 실측과 반대임을 확인하고 문서까지 정정.
