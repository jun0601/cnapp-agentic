# 💰 비용 최적화 전략 (Cost Strategy)

> **한 줄:** 이 프로젝트는 *"무료 크레딧과 평가판 안에서 프로덕션급 멀티클라우드 CNAPP 아키텍처를 증명한다"* 를 **명시적 설계 제약**으로 삼았다. 아래는 비용을 "돈이 없어서 감수한 것"이 아니라 **의도적으로 고른 엔지니어링 트레이드오프**로 다룬 기록이다.

---

## 0. TL;DR

| 레버 | 무엇을 | 절감 효과 |
|---|---|---|
| **Mock-first 개발** | 로직 90%를 `contracts/` mock으로 인프라 0원에 개발·검증 | 빌드 기간 대부분 **AWS 지출 $0** |
| **apply → test → destroy** | 종량제 리소스를 상시 방치하지 않고 데모/검증 시간만 | EKS·RDS 등을 **월 과금 → 시간 과금**으로 |
| **관리형 대신 경량 대체** | NAT Gateway→NAT Instance, Aurora→RDS t3.micro, OpenSearch→pgvector | 고정비 **월 $60+ → 월 $5 수준** |
| **모델 티어링 + 게이트** | Bedrock Haiku/Sonnet 분리 + Triage 게이트로 풀 LLM 루프 최소화 | 토큰 비용을 **소수 케이스로 한정** |
| **Azure = 평가판 + 신원만** | M365 Business 평가 테넌트, 데이터는 AWS S3 전용 | Azure 유료 구독 **$0**, 중복 저장 0 |
| **거버넌스 회피** | Organizations/Identity Center/Control Tower **안 켬** | 무료 크레딧 소멸 트리거 회피 |

**결과 목표:** 무료 티어 + 스타트업/이벤트 크레딧 한도 안에서 전 데모를 완주.

---

## 1. 왜 비용을 "설계 원칙"으로 두었나

멀티클라우드 CNAPP은 원래 **돈으로 밀어붙이면 쉬운** 아키텍처다 — 관리형 서비스를 다 켜고, 클러스터를 상시 띄우고, 벡터DB를 따로 두면 된다. 하지만 그건 **엔지니어링 역량을 증명하지 못한다.**

그래서 이 프로젝트는 반대로 갔다: **"최소 비용에서 같은 아키텍처를 성립시키려면 무엇을 어떻게 바꿔야 하는가"** 를 매 결정의 질문으로 삼았다. 클라우드/MSP 실무에서 FinOps는 선택이 아니라 기본기이기 때문이다.

---

## 2. 결정 원장 (Decision Ledger)

각 결정마다 **무엇을 골랐고 / 얼마를 아꼈고 / 무엇을 포기했나(트레이드오프)** 를 남긴다. 트레이드오프 없는 절감은 없다.

### 2.1 개발 프로세스 — Mock-first (가장 큰 레버)

| | 내용 |
|---|---|
| **결정** | 컴포넌트 간 이음새를 `contracts/*.json`(스키마 7종 + control 카탈로그 14종 + 골든 mock)으로 먼저 고정하고, 엔진·콘솔·attack-path 로직을 **실 인프라 없이** mock 데이터로 개발·검증 |
| **절감** | 빌드 기간의 **대부분을 AWS 지출 $0**로. 실전환은 리스크 큰 한 경로(엔진 실 tool-use)만 골라서 |
| **트레이드오프** | mock ≠ 완전 검증. 실 API의 미지수(모델 ID·리전·권한 경계)는 별도 vertical slice로 실검증 필요 |
| **효과** | 2인이 직렬 의존 없이 병렬 개발 → 시간·비용 동시 절약 |

> 이건 단순한 "테스트 더블"이 아니라 **비용·일정·협업을 동시에 푸는 메타 전략**이다. 계약을 SSOT로 두면 목업↔실물 스왑이 매끄럽다.

### 2.2 컴퓨트 — 관리형 대신 경량 대체

| 결정 | 대안 대비 | 절감(개념) | 트레이드오프 |
|---|---|---|---|
| **NAT Instance** (`t4g.nano` + fck-nat AMI) | NAT Gateway | 월 **~$32 → ~$3** + 데이터 처리비 | HA 없음(단일 인스턴스), 수동 관리. 데모엔 무관 |
| **S3·DynamoDB Gateway Endpoint** | NAT 경유 트래픽 | 두 서비스 트래픽은 **NAT 우회(무료 경로)** | Gateway Endpoint 지원 서비스에 한정 |
| **EKS spot `t3.small` · scale 0~2** | on-demand·상시 노드 | spot 할인 + **비데모 시 노드 0개**(scale-to-zero) | spot 회수 가능성, 콜드스타트. 데모 시 desired만 올림 |
| **RDS `db.t3.micro` + pgvector** | Aurora Serverless / OpenSearch | 벡터DB를 **기존 postgres에 동거**(추가 인프라 0) | 소규모 전용, 스케일 한계. 데모 코퍼스엔 충분 |

> ⚠️ **정직한 주의:** EKS control plane은 노드가 0개여도 **시간당 ~$0.10 고정**이다. 그래서 클러스터를 상시 방치하지 않고 **`apply → test → destroy` 사이클**로 실사용 시간만 과금한다 — 이 규율이 없으면 위 절감이 무의미해진다.

### 2.3 AI — 모델 티어링 + 제품에 박힌 비용 통제

| | 내용 |
|---|---|
| **결정** | Bedrock 단계별 모델 분리 — Triage·Hypothesis·Evidence = **Haiku(저가)**, Reasoning 내러티브 = **Sonnet(고품질)**. 임베딩은 Titan v2(서울) |
| **★ 게이트** | **Triage 게이트가 비용 통제를 제품 로직으로 내재화** — finding 1000건이라도 `severity_id≤2 OR attack_path_id!=null`인 소수만 풀 LLM 루프(Hypothesis→Evidence)로 승급. 나머지는 값싼 캐시 설명에서 멈춤 |
| **절감** | 토큰 비용이 finding 총량이 아니라 **escalate된 소수**에 비례 |
| **트레이드오프** | 게이트 임계값 튜닝 필요(과탐/미탐). 대신 실제 SOC의 트리아지 개념과 정합 |

> 비용 절감이 "인프라 꼼수"가 아니라 **제품 설계의 일부**라는 점이 핵심 — 실서비스도 이렇게 만든다.

### 2.4 멀티클라우드 — 평가판 + 역할 분리

| | 내용 |
|---|---|
| **결정** | Azure는 **M365 Business 평가판 테넌트**(`cnappagentic.onmicrosoft.com`) — Entra ID/CIEM 점검을 유료 Azure 구독 없이. **데이터(회원 PII)는 AWS S3 전용** |
| **절감** | Azure 유료 구독 **$0**. 데이터를 양쪽에 중복 저장하지 않아 **저장비·전송비 0** |
| **트레이드오프** | 평가판 한도·기간. Azure 리소스형 점검은 제한적(신원 중심으로 스코프) |
| **부수 효과** | "AWS=워크로드 / Azure=신원"이라는 **역할 분리가 비용뿐 아니라 아키텍처를 더 깨끗하게** 만듦 — 비용 제약이 설계를 개선한 사례 |

### 2.5 계정 거버넌스 — 크레딧 트리거 회피

| | 내용 |
|---|---|
| **결정** | **단일 계정·단일 리전(서울)**. Organizations·Identity Center·Control Tower **절대 안 켬** |
| **이유** | 이들은 일부 무료 크레딧의 **소멸 트리거**이자 종량 과금 유발. 데모 규모엔 과함 |
| **절감** | 크레딧 보존 + 멀티계정 관리 오버헤드 0 |
| **트레이드오프** | 실무의 계정 분리(prod/dev)·SSO 거버넌스는 범위 밖(확장 훅으로만 명시) |
| **보안 보완** | 계정이 하나여도 **키리스(OIDC/IRSA)·read-only first·HITL 승인**으로 보안 기준은 유지 |

### 2.6 종량제 서비스 — 데모만 켜고 destroy

| | 내용 |
|---|---|
| **결정** | Config·Security Hub·Inspector·Macie·Defender 등 종량제는 **데모/검증 기간에만 켜고 `destroy`** |
| **절감** | 특히 **Macie(스캔 GB당)·Inspector**를 상시 켜지 않아 큰 폭 절감 |
| **트레이드오프** | 연속 모니터링 데이터는 안 쌓임(데모는 스냅샷). 실무는 상시 필요 |
| **가드레일** | AWS Budgets 알림 상시 유지로 초과 조기 감지 |

## 3. 예상 비용 (Order of Magnitude)

정확한 청구가 아니라 **비용 구조의 형태**를 보여주기 위한 추정. 핵심은 "상시 vs 시간 과금"의 차이다.

| 항목 | 상시 방치 시 (안 함) | 이 전략(apply→destroy) |
|---|---|---|
| EKS control plane | ~$73/월 (시간당 $0.10) | 실사용 시간만 (예: 20h ≈ **$2**) |
| NAT | NAT GW ~$32/월 | NAT Instance ~**$3/월** or 시간만 |
| RDS t3.micro | ~$12/월 상시 | 실사용 시간만 (~**$0.017/h**) |
| Bedrock | — | escalate 소수 × Haiku ≈ **수 달러** |
| S3·데이터 | — | **센트 단위** |
| Azure | 유료 구독 시 $수십 | 평가판 = **$0** |

> **결론:** 상시 방치하면 월 $130+가 나올 스택을, **mock-first + destroy 규율 + 경량 대체**로 **무료 크레딧·티어 한도 안**으로 눌렀다.

## 4. 정직한 트레이드오프 (What We Gave Up)

> 절감만 나열하고 대가를 숨기면 신뢰를 잃는다. 아래는 **의식적으로 포기한 것**들이다.

- **고가용성(HA) 없음** — NAT 단일 인스턴스, 단일 AZ 편중, EKS 소규모. **데모 전제**이며 프로덕션이면 다중 AZ·NAT GW·노드 그룹 이중화가 필요.
- **연속 모니터링 아님** — 종량제를 껐다 켜므로 posture는 스냅샷. 실 CNAPP는 상시 수집.
- **Mock ≠ 실검증** — 로직 형태는 검증되나, 실 API의 미지수(Bedrock 모델 ID·리전·권한 경계)는 **vertical slice 실검증**으로 별도 증명해야 함.
- **스케일 한계** — t3.micro pgvector·소규모 노드는 데모 코퍼스/부하 전용. 대량이면 Aurora·OpenSearch·큰 노드로 승급.
- **거버넌스 축소** — 단일 계정이라 멀티계정 분리·조직 SCP 등 엔터프라이즈 거버넌스는 범위 밖.

**이 트레이드오프를 명확히 아는 것 자체가 설계 성숙도의 증거다** — "무엇을 언제 승급해야 하는가"의 판단이 서 있다는 뜻이므로.

---

## 5. 한 줄 회고

> 제약(무료 크레딧·평가판)은 **족쇄가 아니라 렌즈였다.** "돈으로 못 밀 때 무엇을 바꿔야 같은 아키텍처가 성립하나"를 강제로 묻게 했고, 그 답(mock-first·경량 대체·게이트 내재화·역할 분리)이 오히려 **설계를 더 명료하게** 만들었다.
